# OpenClaw 内部数字员工管理后台产品背景与框架

## 一、文档说明

本文档面向我们内部开发团队，用于统一 OpenClaw 当前项目的产品目标、范围边界、核心场景、能力分层和实施顺序。

这不是一份面向外部客户的企业解决方案文档，而是一份面向我们自己团队的内部产品设计文档。当前项目的目标不是做“对外企业 AI 平台”，而是为我们自己打造一套可持续运营的数字员工管理后台。

本文档重点回答以下问题：

1. 我们当前要做的产品到底是什么。
2. 这套产品首先服务哪些内部角色。
3. 数字员工的创建、配置、运行、审计和迭代应如何组织。
4. MVP 应该先做哪些能力，不应该过早做哪些能力。

## 二、项目背景与当前约束

### 2.1 项目定位

当前项目的定位是：

- 为我们自己内部团队建设数字员工管理后台
- 以 OpenClaw 为核心运行底座
- 用管理后台统一管理 Agent、Skill、知识、任务、调度和执行记录
- 先解决内部生产效率和可控性问题，再考虑对外能力抽象

### 2.2 当前阶段约束

当前已知约束如下：

- 我们是初创团队，优先级必须集中，不能同时铺太多外部场景。
- 当前没有外部业务系统接入的刚性要求，不以 CRM、ERP、OA 对接为 MVP 前提。
- 有一台可作为主要部署目标的本地服务器：`192.168.31.189`
- 代码版本管理目标仓库为 [OpenclawTeam](https://github.com/Hooooz/OpenclawTeam.git)
- 团队已经补充了较多 Skill 和一套主 Agent - 子 Agent 协作开发流程，产品与研发设计都应与这套方式兼容

### 2.3 我们当前真正要解决的问题

当前我们不是缺少一个“会聊天的 Agent”，而是缺少一套能让数字员工被稳定管理和持续运营的后台。

核心问题包括：

| 问题 | 当前表现 | 产品要求 |
|---|---|---|
| 数字员工缺少统一管理面 | Agent 配置、能力、运行状态分散 | 需要一个统一后台 |
| Skill 增长后缺少编排与治理 | Skill 多了以后难以知道谁能用、怎么用 | 需要 Skill 注册、分组、授权和调用记录 |
| 执行过程缺少可见性 | 不知道某个任务是谁触发、跑到了哪一步 | 需要任务状态、日志和审计视图 |
| 自动化缺少管理闭环 | 调度、执行、失败、补跑没有统一入口 | 需要调度中心和运行台 |
| 研发协作缺少产品化承载 | 主 Agent / 子 Agent 流程有了，但后台没有沉淀成系统能力 | 需要把协作流程映射为产品模块 |

## 三、产品目标

### 3.1 总目标

把 OpenClaw 从“可运行的 Agent 能力集合”收敛成“内部数字员工管理后台”。

这套后台应至少覆盖以下 6 类能力：

1. 数字员工管理
2. Skill 管理
3. 任务与调度管理
4. 知识与记忆管理
5. 执行记录与审计管理
6. 系统配置与部署运维管理

### 3.2 阶段目标

#### MVP 阶段

建立最小可用后台，让团队能完成：

- 创建数字员工
- 给数字员工绑定 Skill
- 发起一次任务运行
- 查看执行状态和结果
- 配置基础调度
- 查看失败记录

#### 扩展阶段

补齐：

- 知识库接入
- 任务模板
- 权限分级
- 更强的执行回放与审计
- 更稳定的服务器部署和升级能力

#### 平台化阶段

在内部跑稳之后，才考虑：

- 更细颗粒度的多团队隔离
- 更丰富的场景模板
- 可复制的部署包
- 对外标准化能力抽象

## 四、目标用户与内部角色

### 4.1 超级管理员 / 创始人

关注点：

- 系统是否稳定
- 哪些数字员工在跑
- 哪些任务最有价值
- 哪些失败最影响结果

需要的能力：

- 全局看板
- Agent 运行总览
- 调度总览
- 风险与异常总览

### 4.2 运营负责人

关注点：

- 哪些数字员工正在承担稳定的日常工作
- 任务模板是否可复用
- 失败任务如何补跑

需要的能力：

- 数字员工列表
- 任务执行台
- 调度管理
- 历史运行记录

### 4.3 Skill / 工具维护者

关注点：

- 新 Skill 如何注册
- 哪些 Agent 在使用某个 Skill
- Skill 失败率和调用边界

需要的能力：

- Skill 注册中心
- Skill 配置面板
- Skill 调用统计
- Skill 授权边界

### 4.4 知识维护者

关注点：

- 哪些资料已经被接入
- 哪些知识对哪些 Agent 可见
- 文档更新后是否生效

需要的能力：

- 知识库列表
- 知识源同步
- 知识权限控制
- 引用与更新时间追踪

### 4.5 开发者

关注点：

- 主 Agent / 子 Agent 的规格、实现、验证链路是否能被系统承载
- 新功能如何接入后台
- 如何在本地和服务器上快速部署与回归

需要的能力：

- 清晰的模块边界
- 统一的代码目录
- 可视化调试入口
- 明确的测试与验收口径

## 五、产品定义

### 5.1 什么是“数字员工管理后台”

数字员工管理后台不是单个 Agent 的聊天界面，而是一个面向内部运营的控制平面。

它管理的是以下对象：

- 数字员工
- Skill
- 知识源
- 任务模板
- 调度任务
- 执行记录
- 审计日志
- 系统配置

### 5.2 产品边界

#### 当前要做的

- 统一数字员工的注册、配置、启停和查看
- 统一 Skill 的注册、绑定和授权
- 统一任务执行和调度管理
- 统一知识与记忆挂载
- 统一日志、回执和审计可视化
- 统一部署和运行状态管理

#### 当前不优先做的

- 对外客户交付版多租户平台
- 大规模外部系统连接器市场
- 重型 BI 替代
- 高风险全自动外部业务写回
- 一开始就把所有 Skill 和流程都做成高度通用平台

## 六、产品总体框架

### 6.1 总体思路

产品总图应从“后台如何管理数字员工生命周期”来组织，而不是从通用企业系统接入来组织。

建议按六层理解。

### 6.2 六层框架

#### 6.2.1 管理入口层

- Web 管理后台
- 内部调试页面
- API 管理入口

这是团队使用最频繁的入口，MVP 应优先以 Web 管理后台为主。

#### 6.2.2 数字员工管理层

- Agent 列表
- Agent 配置
- Prompt / 角色设定
- Skill 绑定
- 启停状态
- 默认执行策略

这一层负责定义“这个数字员工是谁、会什么、能做什么”。

#### 6.2.3 任务与调度层

- 手动任务发起
- 周期调度
- 队列执行
- 重试与补跑
- 状态跟踪

这一层负责定义“何时执行、怎么执行、失败后怎么处理”。

#### 6.2.4 能力与知识层

- Skill 注册中心
- Skill 分类与版本
- 知识库接入
- 记忆管理
- 场景模板

这一层负责定义“数字员工能调用哪些能力，基于哪些知识做判断”。

#### 6.2.5 运行与审计层

- 执行日志
- trace 视图
- 输入输出摘要
- 错误记录
- 审计记录

这一层负责定义“数字员工到底做了什么、是否可追溯、出了问题如何定位”。

#### 6.2.6 系统与部署层

- 服务器状态
- 部署版本
- 配置中心
- 存储与备份
- 健康检查

这一层负责定义“系统是否跑得住、是否能更新、是否可恢复”。

## 七、核心产品场景

### 7.1 场景一：创建并上线一个数字员工

目标：

- 从后台完成一个新数字员工的创建、配置、Skill 绑定和上线

最小流程：

1. 新建 Agent
2. 配置名称、角色、默认提示词
3. 绑定 Skill
4. 绑定知识源
5. 设定执行方式
6. 保存并启用

### 7.2 场景二：发起一次任务执行

目标：

- 从后台手动触发一个任务，观察完整执行链路

最小流程：

1. 选择 Agent
2. 选择任务模板或输入指令
3. 发起执行
4. 查看中间步骤与最终结果
5. 失败时支持补跑或重新配置

### 7.3 场景三：配置周期调度

目标：

- 让数字员工自动执行日常工作

典型任务：

- 每日汇总
- 定时抓取与整理
- 周报生成
- 指定目录内容处理

### 7.4 场景四：管理 Skill 生命周期

目标：

- 让 Skill 从“零散能力”变成“可治理资产”

核心动作：

- 注册 Skill
- 描述输入输出
- 限定适用 Agent
- 记录调用日志
- 统计成功率和失败率

### 7.5 场景五：管理知识与记忆

目标：

- 让数字员工拥有可控知识输入和可控记忆边界

核心动作：

- 添加知识源
- 指定可见范围
- 触发更新
- 查看引用来源
- 清理过期记忆

### 7.6 场景六：审查一次失败执行

目标：

- 快速定位失败发生在哪一层

需要的视图：

- 请求入口
- Agent 规划过程
- Skill 调用链
- 错误回执
- 最终失败原因

## 九、与现有研发流程的关系

### 9.1 产品与主 Agent - 子 Agent 模式对齐

当前我们已有一套主 Agent - 子 Agent 协作产品开发模式。这个后台不应绕开这套流程，而应逐步承载它。

对应关系建议如下：

| 研发流程对象 | 后台中的产品对象 |
|---|---|
| 主 Agent 产出的规格 | 任务模板 / 产品文档 / 技术方案 / 测试方案 |
| 子 Agent 的实现任务 | 执行任务单 / 任务运行记录 |
| 主 Agent 的最终验证 | 验收记录 / 回归记录 |
| Skill 编组 | Skill Registry / Agent 绑定关系 |

### 9.2 Skill 应纳入后台治理

随着 Skill 变多，后台至少要支持：

- Skill 列表
- Skill 描述
- Skill 所属分类
- Skill 适用场景
- Skill 调用权限
- Skill 最近调用结果

## 十、系统接入与部署原则

### 10.1 当前接入原则

当前阶段不以外部业务系统接入为前提，而以内部可控闭环优先。

MVP 首先接入：

- 本地文件与文档目录
- 内部知识源
- 后台操作入口
- 调度与任务队列
- 系统运行日志

### 10.2 当前部署原则

部署目标优先按单机内部部署设计，主节点为本地服务器 `192.168.31.189`。

设计原则：

- 先单机容器化，再按需要拆服务
- 先内部可运行、可升级、可回滚，再追求复杂分布式
- 先保证日志、备份和恢复，再扩更多功能

### 10.3 版本管理原则

版本管理统一指向 [OpenclawTeam](https://github.com/Hooooz/OpenclawTeam.git)。

建议遵循：

- 文档、代码、部署配置统一入库
- 使用明确的模块目录管理后台前后端和部署文件
- 重要里程碑必须可回溯到 Git 提交

## 十一、版本路线

### 11.1 MVP

MVP 只做以下闭环：

- 管理后台基础框架
- Agent 注册与配置
- Skill 绑定
- 手动任务执行
- 基础调度
- 执行日志查看

### 11.2 V1

在 MVP 基础上补齐：

- 知识源管理
- 记忆管理
- 权限分级
- 更好的失败排查
- 更好的服务器部署与监控

### 11.3 V2

进一步补齐：

- 任务模板体系
- 更强的调度中心
- 更细粒度的审计体系
- 更完整的运维控制面
- 平台化扩展接口

## 十二、当前优先级建议

### 12.1 应优先做的

1. 管理后台基础骨架
2. Agent Registry
3. Skill Registry
4. Task Run / Scheduler
5. Execution Log / Audit 基础视图

### 12.2 应暂缓的

1. 面向外部客户的通用化包装
2. 重型多租户
3. 大量外部系统连接器
4. 高风险自动化写回
5. 复杂 BI 和报表平台化

## 十三、结论

当前项目的正确方向，不是继续把 OpenClaw 写成“泛企业方案”，而是把它收敛成一套我们自己可持续使用的内部数字员工管理后台。

只要我们先把 Agent、Skill、任务、调度、知识、执行记录和部署运维这几条主线打通，后续不论是内部扩展，还是再抽象成对外产品，都会更稳。
