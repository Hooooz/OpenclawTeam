# OpenClaw 内部数字员工管理后台阶段总结与测试用例草案

## 一、文档说明

本文档面向我们内部开发团队，用于定义当前项目的阶段划分、每阶段交付边界、阶段评审方式，以及适用于管理后台项目的最小测试骨架。

这份文档重点服务两个目的：

1. 帮团队判断“当前阶段是否真的完成”。
2. 帮主 Agent 在后续实现阶段有一套统一的验证和回灌口径。

## 二、阶段划分原则

### 2.1 阶段按内部可用价值划分

每个阶段都必须形成团队内部可感知的使用价值，而不是只完成某个技术模块。

### 2.2 阶段按控制面与运行面闭环划分

本项目的关键不是页面做完，而是：

- 后台能配
- 运行时能跑
- 日志能看
- 失败能定位

### 2.3 阶段评审必须看证据

阶段完成不能靠口头说明，必须看：

- 页面或接口实测
- 运行记录
- 日志截图或明细
- 通过的测试项

## 三、阶段划分建议

### 3.1 P0：方案与工程初始化

阶段目标：

- 明确产品和技术边界
- 建立仓库与目录结构
- 明确部署目标和最小工程方案

交付项：

- 产品背景与框架
- 技术方案与开发清单
- 测试用例草案
- 基础工程目录

完成标志：

- 文档冻结到可执行程度
- 仓库位置明确
- 初始工程可以开始开发

### 3.2 P1：后台骨架与对象管理

阶段目标：

- 建立管理后台基础骨架
- 完成 Agent 和 Skill 的可视化管理

交付项：

- Dashboard
- Agent 列表 / 创建 / 编辑
- Skill 列表 / 查看
- 基础权限框架

完成标志：

- 团队能通过后台维护 Agent 和 Skill 基础信息

### 3.3 P2：任务执行闭环

阶段目标：

- 打通从后台发起任务到查看结果的最小闭环

交付项：

- 手动任务执行
- Run Record 列表
- Run Detail 页面
- 基础错误展示

完成标志：

- 至少一个 Agent 可以从后台被手动触发并返回结构化结果

### 3.4 P3：调度、日志与失败处理

阶段目标：

- 打通自动化执行和失败排查闭环

交付项：

- Scheduler CRUD
- Queue / Worker
- Retry 机制
- Audit / Trace 基础视图

完成标志：

- 至少一个周期任务可自动执行
- 失败可被定位并可重试

### 3.5 P4：知识与系统运维

阶段目标：

- 补齐知识接入和系统运行能力

交付项：

- Knowledge Source 管理
- Memory 策略
- Health Check
- Backup 基础能力

完成标志：

- 知识源可维护
- 系统健康状态可见
- 备份和恢复路径明确

### 3.6 P5：本地服务器部署与验收

阶段目标：

- 在 `192.168.31.189` 上形成稳定部署版本

交付项：

- 部署文件
- 环境变量模板
- 初始化脚本
- 发布说明
- 回滚说明

完成标志：

- 能完成一次完整部署
- 能完成一次升级
- 能完成一次最小回滚验证

## 四、阶段总结模板

每个阶段统一按下面模板沉淀：

### 4.1 阶段目标

- 本阶段要完成什么
- 本阶段明确不做什么

### 4.2 已完成内容

- 页面
- 接口
- Runtime 能力
- 部署能力

### 4.3 当前问题

- 功能缺口
- 性能问题
- 部署问题
- 权限问题

### 4.4 验证证据

- 已通过测试项
- 实机演示记录
- 关键截图或日志位置

### 4.5 下一阶段建议

- 继续做什么
- 应暂缓什么

## 五、阶段评审模板

每个阶段评审至少回答：

1. 后台配置是否真的可用。
2. 运行时是否真的跑通。
3. 日志和错误是否真的可定位。
4. 当前是否具备进入下一阶段的条件。
5. 哪些问题不能带到下一阶段。

评审结论统一为：

- 通过
- 补齐后通过
- 不通过

## 六、测试策略

### 6.1 测试主线

当前项目的测试主线是：

1. 管理后台对象管理正确
2. Runtime 执行链路正确
3. 调度和重试正确
4. 日志与审计正确
5. 本地服务器部署可重复

### 6.2 测试分层

| 层级 | 目标 | 例子 |
|---|---|---|
| 冒烟测试 | 确认服务活着 | 管理后台可打开，API 可响应 |
| 功能测试 | 确认页面和接口正确 | 创建 Agent、绑定 Skill |
| 集成测试 | 确认控制面和运行面打通 | 后台发起任务，Runtime 执行 |
| E2E 测试 | 确认关键业务闭环 | 创建 Agent -> 发起任务 -> 查看结果 |
| 部署验收测试 | 确认服务器部署稳定 | 首次部署、升级、回滚 |

## 八、E2E 样板场景

### 8.1 场景一：创建数字员工并执行一次任务

步骤：

1. 登录后台。
2. 创建一个 Agent。
3. 绑定至少一个 Skill。
4. 手动发起任务。
5. 查看结果。

验收点：

- Agent 创建成功
- Skill 绑定成功
- Run Record 正常生成
- 最终结果可见

### 8.2 场景二：配置一个周期任务

步骤：

1. 选择 Agent。
2. 创建一个调度计划。
3. 等待触发或手动模拟触发。
4. 查看生成的运行记录。

验收点：

- 调度记录正确保存
- 任务按计划触发
- 状态可追踪

### 8.3 场景三：定位一次失败执行

步骤：

1. 构造一个失败任务。
2. 在后台打开 Run Detail。
3. 查看 trace、错误和失败步骤。
4. 执行补跑。

验收点：

- 失败原因可见
- 补跑操作可用
- 新旧 run 记录可区分

## 九、性能与稳定性测试建议

### 9.1 当前重点

当前不急于做重压测试，先关注：

- 后台页面响应是否正常
- 单次任务执行是否稳定
- 调度任务是否稳定
- 失败重试是否可控

### 9.2 初版指标建议

- 后台关键页面首屏时间
- API P95 响应时长
- Run 成功率
- 调度成功率
- Retry 成功率

## 十、上线前验收建议

### 10.1 上线前必须满足

- 后台核心页面可用
- Agent 和 Skill 的基础管理可用
- 手动任务执行可用
- 调度至少一个样板任务可用
- 审计和日志基本可查
- 部署文档完整

### 10.2 当前不建议上线的情况

- 没有 run record
- 没有错误日志
- 调度失败不可见
- 没有最小回滚方式
- 页面能配但 Runtime 实际跑不起来

## 十一、统一问题收集建议

按你的要求，后续研发过程中应尽量不打断主流程。建议统一维护“待确认问题清单”，只在以下情况集中提问：

- 阻塞代码实现
- 阻塞部署
- 阻塞验收标准

非阻塞问题统一先记录，不立即中断。

## 十二、结论

这份测试和阶段文档的价值，不是把测试写得很大，而是先守住管理后台项目最关键的五条线：对象管理、任务执行、调度运行、日志审计、服务器部署。

只要这五条线稳定，数字员工管理后台才真正具备内部长期使用价值。
